# Skill Invocation Patterns

This pipeline supports two approaches for running agents with skills:

## 1. Direct Skill Invocation

Let Claude discover and invoke skills automatically via the SDK's Skill tool.

### When to Use
- Simple prompts that don't need heavy templating
- Skills that work well with argument passing
- When you want Claude to autonomously choose the skill

### How It Works

```python
# Enable Skills in the agent
result = await run_agent(job, log_dir, model, enable_skills=True)
```

This:
1. Adds `"Skill"` to `allowed_tools`
2. Sets `setting_sources=["user", "project"]` to load skills from filesystem
3. Claude discovers skills from `.claude/skills/*/SKILL.md`
4. Claude invokes skills when descriptions match the prompt

### Example: discover-components

```python
# Simple prompt that triggers the skill
prompt = """Use the discover-components skill to discover platform components.

Arguments:
--platform=aap
--checkouts-dir=./checkouts/ansible
--entry-repo=awx-operator
"""

job = {"name": "discover-aap", "cwd": ".", "prompt": prompt}
result = await run_agent(job, log_dir, "sonnet", enable_skills=True)
```

Claude sees the prompt, matches it to the `discover-components` skill description, and invokes it with the provided arguments.

### Advantages
- Less code in phases.py
- Skills are self-contained and reusable
- Claude handles skill selection and invocation
- Easier to test skills independently

### Disadvantages
- Less control over prompt construction
- Can't inject complex runtime context easily
- Relies on skill description matching

## 2. Templated Prompts (Extracted Instructions)

Manually extract instructions from SKILL.md and template with runtime data.

### When to Use
- Complex workflows needing context injection
- When you need full control over the prompt
- Runtime data that must be injected (git metadata, build info, etc.)

### How It Works

```python
# Read skill file
skill_content = Path(".claude/skills/my-skill/SKILL.md").read_text()

# Extract instructions
instructions_match = re.search(r'## Instructions\n\n(.+)', skill_content, re.DOTALL)
instructions = instructions_match.group(1).strip()

# Template with runtime data
git_context = get_git_metadata(...)
build_info = get_build_info(...)

prompt = f"""Generate architecture for this component.

{git_context}
{build_info}

{instructions}
"""

job = {"name": "component-a", "cwd": "/path/to/repo", "prompt": prompt}
result = await run_agent(job, log_dir, "opus", enable_skills=False)  # Default
```

### Example: generate-architecture

```python
# Pre-gather context
git_context = _format_git_context(component.checkout_path)
build_context = format_build_info_context(build_info)
kustomize_context = format_kustomize_context(kustomize_ctx)

# Template the prompt
prompt = f"""Generate a comprehensive architecture summary for this component.

Distribution: {distribution}
Repository: {component.repo_org}/{component.repo_name}
Manifests folder: {component.source_folder}

{build_context}
{kustomize_context}
{git_context}

IMPORTANT: For "Generated By", use: {model_display}

{instructions}
"""

job = {"name": component.key, "cwd": str(component.checkout_path), "prompt": prompt}
result = await run_agent(job, log_dir, model, enable_skills=False)
```

### Advantages
- Full control over prompt structure
- Can inject complex runtime context
- Pre-gather data so agent doesn't need to
- Explicit about what gets included

### Disadvantages
- More code in phases.py
- Need to manually extract instructions
- Harder to reuse across different contexts

## Current Usage in Pipeline

| Phase | Method | Reason |
|-------|--------|--------|
| `discover-components` | **Direct Skill** | Simple args, no heavy context |
| `generate-architecture` | **Templated** | Injects git, build, kustomize context |
| `collect-architectures` | Could use either | Currently uses script, could be skill |
| `generate-platform-architecture` | **Templated** | Injects build context, entry component |
| `generate-diagrams` | **Templated** | Specifies output formats and paths |

## Choosing an Approach

**Use Direct Skill Invocation when:**
- Prompt is simple and mostly static
- Skill arguments are straightforward
- No complex runtime data to inject
- Want Claude to autonomously select skill

**Use Templated Prompts when:**
- Need to inject complex runtime context
- Pre-gathering data improves efficiency
- Want explicit control over what's included
- Prompt structure is dynamic based on data

## Implementation Details

Both approaches use the same `run_agent()` function:

```python
async def run_agent(
    job: dict,
    log_dir: Path,
    model: str = "sonnet",
    enable_skills: bool = False  # Toggle between approaches
) -> dict:
```

The `enable_skills` flag controls:
- Whether `"Skill"` is in `allowed_tools`
- Whether `setting_sources=["user", "project"]` is set
- Whether Claude can discover and invoke skills

## Converting Between Approaches

**From Templated → Direct Skill:**

```python
# Before (templated)
instructions = extract_instructions_from_skill("my-skill")
prompt = f"Do X with Y\n\n{instructions}"
result = await run_agent(job, log_dir, model)

# After (direct skill)
prompt = "Use my-skill with --arg=Y"
result = await run_agent(job, log_dir, model, enable_skills=True)
```

**From Direct Skill → Templated:**

```python
# Before (direct skill)
prompt = "Use my-skill with --arg=Y"
result = await run_agent(job, log_dir, model, enable_skills=True)

# After (templated)
instructions = extract_instructions_from_skill("my-skill")
runtime_data = gather_context(...)
prompt = f"{runtime_data}\n\n{instructions}"
result = await run_agent(job, log_dir, model)
```

## Best Practices

1. **Start with Direct Skill** - Try direct invocation first, it's simpler
2. **Add Templating When Needed** - Switch to templated if you need context injection
3. **Document the Choice** - Comment why you chose each approach
4. **Keep Skills Reusable** - Write skills that work well with both approaches
5. **Test Both Ways** - Some skills work better with one approach than the other
